# Практическое занятие №1

## REST API для каталога учебных курсов

Проект реализует REST API с использованием Node.js и Express, следуя принципам middleware-конвейера. Архитектура построена вокруг трёх ключевых компонентов: logger (журнал запросов), timer (время выполнения запроса) и errorHandler (обработка ошибок). Существуют дополнительные компоненты: router (реализация точек доступа) и model (структура данных и валидация).

Для хранения данных используется массив в памяти процесса. Валидация самих данных происходит в модели и проверяет 4 правила: название курса не пустое; имя преподавателя не пустое; количество кредитов должно быть положительным числом; вместимость должна быть положительным числом. Если данные не проходят валидацию, то в маршруте создаётся ошибка с кодом 400, которая направляется в общий конвейер обработки ошибок.

API предоставляет 3 стандартные точки доступа и 1 дополнительную:

- `GET /api/items` — возвращает полный список курсов
- `GET /api/items/{id}` — возвращает один курс по идентификатору или ошибку 404
- `POST /api/items` — создаёт новый курс с проверкой входных данных
- `DELETE /api/items/{id}` — удаляет существующий объект по id или ошибку 404

## Проверка базовой функциональности

Первая проверка демонстрирует получение списка всех курсов. При старте приложения в память уже загружены четыре тестовых курса, поэтому запрос к `/api/items` возвращает массив с ожидаемой структурой данных. Успешное выполнение этой проверки подтверждает, что маршрутизация работает, сервер отвечает и данные корректно сериализуются в JSON.

```bash
curl http://localhost:3000/api/items
```

Вторая проверка показывает получение конкретного курса по идентификатору. Поскольку идентификаторы генерируются случайно при каждом запуске, в сценарии проверки предусмотрен шаг извлечения ID из ответа предыдущего запроса.

```bash
curl http://localhost:3000/api/items/<id>
```

Третья проверка показывает создание нового курса.

```bash
curl -X POST http://localhost:3000/api/items \
  -H "Content-Type: application/json" \
  -d '{"title":"Веб-программирование","lecturer":"Зайцева Е.К.","credits":4,"capacity":35}'
```

## Проверка обработки ошибок

Четвёртый сценарий специально запрашивает курс с заведомо несуществующим идентификатором. В ответ сервер должен вернуть статус 404 и объект ошибки в едином формате.

```bash
curl http://localhost:3000/api/items/00000000-0000-0000-0000-000000000000
```

Пятая проверка тестирует валидацию входных данных. При создании курса с пустым названием и именем преподавателя, отрицательным числом кредитов и нулевой вместимостью сервер возвращает статус 400 и сообщение, перечисляющее все нарушения.

```bash
curl -X POST http://localhost:3000/api/items \
  -H "Content-Type: application/json" \
  -d '{"title":"","lecturer":"","credits":-1,"capacity":0}'
```

## Ответы на контрольные вопросы

**1. Что в Вашей работе выступает независимой переменной и что выступает зависимой переменной**

Независимая переменная — порядок следования middleware в конвейере.

Зависимая переменная — время ответа сервера, измеряемое в миллисекундах.

**2. Какая часть решения должна быть в ядре каркаса, а какая должна оставаться на стороне приложения**

В ядре каркаса должны оставаться генерация requestId, конвейерная обработка ошибок с единым форматом и базовое логирование.

На стороне приложения должны остаться бизнес-логика, модели данных, маршруты и правила валидации.

**3. Какие типовые ошибки или антипаттерны Вы обнаружили в процессе реализации**

В первоначальной версии Timer пытался установить заголовок X-Response-Time в событии finish, но Logger уже отправил ответ и закрыл соединение. В первой версии Logger пытался делать слишком много: генерировать ID, логировать запрос, логировать ответ, измерять время. Это нарушало принцип единственной ответственности и создавало риск, что при изменении логики логирования сломается что-то ещё. Разделение на Logger, Timer и Error Handler исправило эту ситуацию. На этапе отладки было удобно возвращать клиенту полный stack trace, но это создавало угрозу безопасности. Пришлось добавить условную логику, скрывающую технические детали.

**4. Как бы изменились выводы, если бы масштаб системы увеличился в десять раз**

Пришлось бы подключать базу данных. Отправлять логи в другое место, чтобы не забивать память. Если бы было несколько серверов, то появилась бы ошибка возникновения одинаковых идентификаторов на разных серверах и пришлось бы делать идентификаторы уникальными для всей системы сразу. При реальной нагрузке пришлось бы ограничить частоту запросов и проверять, кто их посылает.
