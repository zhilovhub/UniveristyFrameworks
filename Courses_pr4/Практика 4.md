# Практическое занятие №4

https://github.com/zhilovhub/UniveristyFrameworks/tree/master/Courses_pr4

## 1. Цель работы

Реализовать учебную веб-службу, моделирующую процесс из нескольких шагов, который выдерживает повторную доставку событий и частичные сбои. Служба должна быть наблюдаемой: по журналам и показателям должно быть понятно, что произошло, на каком шаге и почему.

## 2. Постановка задачи

1. Реализовать машину состояний с четырьмя и более состояниями и переходами по событиям.
2. Хранить текущее состояние в памяти по ключу процесса.
3. Обеспечить идемпотентность по ключу идемпотентности — повторная доставка не меняет состояние.
4. Реализовать одну компенсацию, откатывающую результат шага при сбое следующего шага.
5. В журнал писать переходы, повторы и компенсации со сквозным идентификатором корреляции.
6. Реализовать проверки здоровья: живости и готовности; при критической деградации проверка готовности должна становиться неуспешной.
7. Реализовать показатели: успешные/ошибочные переходы, повторные доставки, компенсации, оценка задержки шагов.

## 5. Описание реализации

### 5.1. Машина состояний

В файле [`src/states.js`](src/states.js:1) определены состояния и события:

| Состояние             | Описание                                                  |
|-----------------------|-----------------------------------------------------------|
| `Новый`               | Процесс только создан, заявка ещё не подана               |
| `ЗаявкаПринята`       | Заявка зарегистрирована                                   |
| `РесурсЗабронирован`  | Переговорка временно зарезервирована                      |
| `ДоступВыдан`         | Сгенерирован код доступа в переговорку                    |
| `Завершён`            | Процесс успешно завершён                                  |
| `Ошибка`              | Зафиксирован сбой, требующий внешнего вмешательства       |
| `КомпенсацияВыполнена`| Бронь была отменена откатом из-за сбоя следующего шага    |

Переходы определяются таблицей в [`src/stateMachine.js`](src/stateMachine.js:13). Каждый переход связан с шагом-обработчиком из [`src/steps.js`](src/steps.js:1):

```
Новый ──ПринятьЗаявку──▶ ЗаявкаПринята
ЗаявкаПринята ──Забронировать──▶ РесурсЗабронирован
РесурсЗабронирован ──ВыдатьДоступ──▶ ДоступВыдан
                                  └─(сбой)─▶ КомпенсацияВыполнена
ДоступВыдан ──Завершить──▶ Завершён
```

### 5.2. Хранилище процессов

[`src/storage.js`](src/storage.js:1) реализует `ProcessStorage` как singleton, поверх `Map<processId, { state, context, idempotencyKeys, history }>`. Контекст шага накапливается между переходами (например, `bookingId`, `accessCode`).

### 5.3. Идемпотентность

В каждом запросе на маршрут `POST /process-event` обязательно присутствует `idempotencyKey`. Если для данного `processId` ключ уже встречался — возвращается сохранённый результат, состояние не меняется, а счётчик `duplicateEvents` инкрементируется. Реализация — в [`src/routes/process.js`](src/routes/process.js:21).

### 5.4. Компенсация

В [`src/compensation.js`](src/compensation.js:1) реализована функция `compensateBooking`. Она вызывается из машины состояний только тогда, когда переход помечен флагом `compensateOnFail` и шаг-обработчик выбросил ошибку. Это переход `РесурсЗабронирован → ВыдатьДоступ`: если доступ выдать не удалось, бронь откатывается и процесс уходит в состояние `КомпенсацияВыполнена`.

Сбой в шаге `ВыдатьДоступ` имитируется флагом, который включается административным эндпоинтом `POST /admin/fail/issue/on`.

### 5.5. Журналирование и корреляция

[`src/logger.js`](src/logger.js:1) пишет JSON-строки в `stdout`. В каждой записи присутствуют:

- `timestamp`,
- `correlationId` (передаётся клиентом),
- `message` (тип события, например, `transition.success`, `event.duplicate`, `compensation.start`),
- произвольные дополнительные поля (`processId`, `from`, `to`, `step`, `error` и т. п.).

### 5.6. Показатели

[`src/metrics.js`](src/metrics.js:1) собирает:

- `transitionsSuccess` — успешные переходы;
- `transitionsError` — ошибочные/недопустимые переходы;
- `duplicateEvents` — повторные доставки;
- `compensations` — выполненные компенсации;
- `stepStats` — для каждого шага: число вызовов, среднее, p50 и максимум задержки;
- `recentLatencies` — окно из 20 последних значений.

Доступ — через `GET /metrics`.

### 5.7. Проверки здоровья

[`src/health.js`](src/health.js:1):

- `GET /health/live` — всегда `200 alive` с временем работы.
- `GET /health/ready` — `200 ready` или `503 not ready`, если включена критическая деградация.
- Деградация переключается административно: `POST /admin/degrade` и `POST /admin/recover`.

### 5.8. Маршруты API

| Метод | Путь                          | Назначение                                        |
|-------|-------------------------------|---------------------------------------------------|
| POST  | `/process-event`              | Подать событие в машину состояний                 |
| GET   | `/process-event/:processId`   | Получить состояние, контекст и историю процесса   |
| GET   | `/metrics`                    | Текущие показатели                                |
| GET   | `/health/live`                | Проверка живости                                  |
| GET   | `/health/ready`               | Проверка готовности                               |
| POST  | `/admin/degrade`              | Включить деградацию                               |
| POST  | `/admin/recover`              | Выключить деградацию                              |
| POST  | `/admin/fail/issue/on`        | Включить имитацию сбоя на шаге `ВыдатьДоступ`     |
| POST  | `/admin/fail/issue/off`       | Выключить имитацию сбоя                           |
| GET   | `/admin/state`                | Текущие административные флаги                    |
| GET   | `/admin/compensation-log`     | Журнал выполненных компенсаций                    |
| GET   | `/admin/processes`            | Список всех процессов в памяти                    |

## 8. Выводы

В рамках практической работы №4 разработана веб-служба бронирования переговорных на Node.js + Express. Полностью реализованы все требования задания:

1. Машина состояний из семи состояний с переходами по четырём бизнес-событиям и одним служебным.
2. Хранение состояния в памяти по ключу процесса с накоплением контекста и истории переходов.
3. Идемпотентность по ключу: повторная доставка возвращает кэшированный результат и не меняет состояние.
4. Компенсация при сбое шага `ВыдатьДоступ`: бронь откатывается, процесс уходит в `КомпенсацияВыполнена`.
5. Структурированное JSON-журналирование со сквозным `correlationId` во всех записях.
6. Проверки живости и готовности; готовность падает в 503 при критической деградации.
7. Метрики: успешные/ошибочные переходы, повторные доставки, компенсации и задержки шагов с агрегатами.

Дополнительно реализованы административные ручки для управления флагами имитации сбоев и деградации, маршруты `/admin/processes` и `/admin/compensation-log` для удобного контроля состояния.
